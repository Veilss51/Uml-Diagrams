## 1. Диаграмма компонентов

### Описание архитектуры
Диаграмма компонентов отображает архитектурное разделение системы на три слоя: клиентский, серверный и хранилища данных. 

- **Клиентский слой** представлен мобильным приложением (iOS/Android) и веб-интерфейсом, которые взаимодействуют с сервером через **API Gateway** (REST API).
- **Серверный слой** включает модули логики (**Order Service**, **Payment Service**, **Delivery Service**, **User Service**), каждый из которых использует соответствующий репозиторий для доступа к данным.
- **Слой данных**: Репозитории обращаются к единой базе данных **PostgreSQL** и кэшу **Redis**.
- **Дополнительные связи**: Модуль заказов использует Модуль оплаты для транзакций, а Модуль доставки использует Внешний API карт для маршрутизации.

Такая структура обеспечивает разделение ответственности, независимое масштабирование сервисов и отказоустойчивость.

## 2. Диаграмма объектов

### Описание экземпляров

Диаграмма объектов иллюстрирует конкретные экземпляры классов системы в определённый момент времени (снимок состояния).

Здесь представлены объекты:

- **Клиент** (`customer1`): id=101, имя «Иван Петров», статус «Active».
- **Заказ** (`order1`): id=5001, статус «Cooking», сумма 1500₽.
- **Блюдо** (`burger1`): id=20, название «Чизбургер», цена 500₽.
- **Платеж** (`pay1`): id=900, статус «Success», метод «Card».
- **Курьер** (`courier1`): id=303, статус «OnDelivery».

Связи между объектами показывают, что `customer1` владеет заказом `order1`, заказ содержит блюдо `burger1` (через промежуточный объект `OrderItem`), заказ оплачен через `pay1` и доставляется `courier1`. Диаграмма демонстрирует реальные данные, которые система обрабатывает во время активного заказа.

## 3. Диаграмма вариантов использования

### Описание функциональности

Диаграмма вариантов использования описывает функциональность системы с точки зрения взаимодействия актёров. Выделены четыре основных актёра: **Гость**, **Зарегистрированный пользователь (Customer)**, **Курьер** и **Администратор**.

- **Гость** может просмотреть меню, зарегистрироваться и войти в систему.
- **Зарегистрированный пользователь**, помимо общих действий, имеет возможность оформить заказ, оплатить его, отследить доставку и оставить отзыв.
- **Курьер** может принять заказ в доставку, обновить местоположение и подтвердить доставку.
- **Администратор** дополнительно может просматривать всех пользователей, управлять меню ресторанов и разрешать споры.

Вариант использования **«Оформить заказ»** включает **«Оплатить заказ»**, а **«Отследить доставку»** расширяет **«Просмотреть профиль»**. Такая структура позволяет чётко разграничить роли и доступный функционал.

## 4. Диаграмма развёртывания

### Описание инфраструктуры
Диаграмма развёртывания показывает физическое распределение компонентов системы по узлам.

- **Клиентский узел**: Смартфон пользователя или браузер. Внутри развёрнут артефакт мобильного приложения (`app-release.apk`) или веб-клиент (`bundle.js`).
- **Серверный узел (Cloud)**: Виртуальный сервер (например, AWS EC2 или Kubernetes Pod). Внутри работает среда выполнения (JVM/Python), веб-сервер Nginx и запущенные микросервисы.
- **Узел данных**: Выделенный сервер СУБД PostgreSQL и сервер кэширования Redis.
- **Связи**: Показаны физические протоколы связи (HTTPS, JDBC, TCP).

# Лабораторная работа №4
## Паттерны проектирования в системе доставки еды FoodFast

В основе архитектуры системы лежат четыре ключевых паттерна проектирования: **Абстрактная фабрика**, **Декоратор**, **Фабричный метод** и **Наблюдатель**. Они обеспечивают гибкость создания заказов, расширение функциональности без изменения существующего кода, адаптацию под разные категории пользователей и реакцию на события доставки. Ниже представлены диаграммы, иллюстрирующие реализацию этих паттернов.

---

## Диаграмма Abstract Factory

Абстрактная фабрика (`OrderFactory`) определяет интерфейс для создания двух типов продуктов: `DeliveryOrder` (заказ с доставкой) и `PickupOrder` (заказ на самовывоз). Конкретные фабрики `ExpressOrderFactory` и `ScheduledOrderFactory` реализуют создание заказов соответствующего типа доставки. Это позволяет легко добавлять новые фабрики (например, для групповых заказов) без изменения клиентского кода. Каждый заказ композиционно включает объект `OrderItem` с атрибутами `quantity`, `price`, `menuItem`.

## Диаграмма Decorator

Декоратор позволяет динамически добавлять заказам новое поведение. Базовый абстрактный класс `Order` содержит общие атрибуты (id, createdAt, totalAmount, status) и методы (calculateTotal, cancel). Конкретные заказы (DeliveryOrder, PickupOrder) наследуются от Order. Абстрактный декоратор OrderDecorator оборачивает объект Order и делегирует ему вызовы. Конкретные декораторы PriorityOrderDecorator (добавляет приоритетную обработку) и PromoOrderDecorator (добавляет применение промокодов) расширяют поведение заказов, не изменяя их исходный код.

## Диаграмма Factory Method

Фабричный метод используется для создания заказов, адаптированных под разные категории пользователей. Абстрактный класс `User` определяет общие свойства и методы. `RegisteredUser` (зарегистрированный пользователь) вводит фабричный метод _createOrder(), который возвращает объект Order. В зависимости от типа пользователя (RegularUser или PremiumUser) этот метод переопределяется и создаёт заказы с разными условиями доставки или скидками. Клиентский метод placeNewOrder() вызывает фабричный метод, получая нужный заказ. Такой подход упрощает добавление новых типов пользователей без изменения существующей логики.

## Диаграмма Observer

Паттерн Наблюдатель реализует механизм оповещения о статусе заказа. `OrderStatus` выступает в роли субъекта (`Subject`): он хранит список наблюдателей, а также информацию о текущем статусе заказа, времени обновления и идентификаторе заказа. При вызове updateStatus() субъект уведомляет всех подписчиков через notify(). Конкретные наблюдатели:  
`NotificationService` – отправляет push/SMS уведомления клиенту;   
`AnalyticsService` – собирает статистику для анализа доставки;  
`CourierDispatcher` – перераспределяет заказы между курьерами.  
Это позволяет легко добавлять новые реакции на события (например, логирование или интеграцию с внешними сервисами) без изменения логики статусов заказа.

## Реализация и развёртывание

Система реализована на `Python`. В текущей версии используется REST API интерфейс, все компоненты работают в распределённой архитектуре. Хранение данных организовано через SPostgreSQL (для масштабирования). Перечисленные паттерны обеспечивают высокую гибкость: легко добавить новые типы заказов (через фабрику), приоритетную обработку или скидки (через декоратор), адаптировать поведение для разных пользователей (через фабричный метод) и расширить систему уведомлений (через наблюдателя). Архитектура готова к переходу на микросервисы с минимальными изменениями.
